**STEP 6A: IDENTIFY APPLICATION INTERACTIONS (Applications Only)**

This step applies ONLY if the component is an APPLICATION (not a library).

Identify how this application interacts with OTHER INTERNAL APPLICATIONS at runtime:

1. **HTTP API Calls to Other Applications:**
   - Use Grep to find HTTP client usage (reqwest, axios, requests, fetch, http.Client)
   - Look for hardcoded URLs, environment variables pointing to other services
   - Identify which endpoints are called, what methods, what data is sent/received

   For Rust (using reqwest):
   ```bash
   grep -rn "reqwest::Client\|client\.get\|client\.post" src/
   grep -rn "http://" src/ | grep -v "localhost"
   grep -rn "https://" src/
   ```

   For Node.js (using axios/fetch):
   ```bash
   grep -rn "axios\.\(get\|post\|put\|delete\)" src/
   grep -rn "fetch(" src/
   ```

   For Python (using requests):
   ```bash
   grep -rn "requests\.\(get\|post\|put\|delete\)" src/
   grep -rn "http\.client" src/
   ```

2. **Shared Database Access:**
   - Look for database connection strings, table/collection names
   - If multiple applications write to the same database/tables, that's shared database access
   - Use Grep to find database operations (SQL queries, ORM usage)
   - Check configuration files for database names/URLs

   ```bash
   grep -rn "CREATE TABLE\|INSERT INTO\|UPDATE.*SET\|DELETE FROM" src/
   grep -rn "db\.\|database\.\|collection\." src/
   ```

3. **Message Queue/Event Bus Communication:**
   - Use Grep to find message queue libraries (RabbitMQ, Kafka, Redis pub/sub, AWS SQS)
   - Identify topics/queues/channels this application publishes to or subscribes from
   - Note which other applications might be consuming these messages

   ```bash
   grep -rn "publish\|subscribe\|send.*queue\|consume" src/
   grep -rn "kafka\|rabbitmq\|redis.*pub\|sqs" src/
   ```

4. **gRPC/GraphQL Calls:**
   - Use Grep to find gRPC client stubs or GraphQL client usage
   - Identify which services are being called

   ```bash
   grep -rn "grpc.*Client\|.*ServiceClient" src/
   grep -rn "graphql\|gql\|apollo" src/
   ```

1. **HTTP Method and Path**

   - What does this endpoint do?
   - When is it used?
   - What business logic does it trigger?

   - Show actual HTTP request format
   - Include required headers (Authorization, Content-Type)
   - Include request body for POST/PUT/PATCH
   - Show path and query parameters in context

4. **Response Example**
   - Show actual JSON/response format
   - Include typical success response (200, 201, etc.)
   - Optionally show common error responses (400, 404, 500)
   - Use realistic example data

5. **Authentication/Authorization**
   - Note if endpoint requires authentication
   - Specify auth method (Bearer token, API key, etc.)

**Extraction Strategy:**

1. **Find Route Definitions:**
   - Use Grep to locate all route declarations
   - Map HTTP methods to paths
   - Identify path parameters, query parameters

2. **Read Handler Functions:**
   - For each route, read the handler implementation
   - Understand what it does (DB query, external call, computation)
   - Identify request body structure from parameter types
   - Identify response structure from return statements

3. **Extract Request/Response Types:**
   - Find struct/class/interface definitions for DTOs
   - Look for JSON serialization tags/decorators
   - Infer types from function signatures

4. **Create Realistic Examples:**
   - Use actual field names from code
   - Use appropriate data types (UUIDs, timestamps, etc.)
   - Show realistic values, not just placeholders

5. **Document in Order:**
   - Group related endpoints together
   - Order by resource (all /users endpoints, then /posts, etc.)
   - Or order by HTTP method (all GETs, then POSTs, etc.)

**Understanding Rust Service Structure:**

**Go Concurrency Patterns:**
```go
// Worker pool pattern
func workerPool(jobs <-chan Job, results chan<- Result) {
    for j := range jobs {
        results <- process(j)
    }
}

// Context cancellation
func doWork(ctx context.Context) error {
    select {
    case <-ctx.Done():
        return ctx.Err()
    case result := <-processData():
        return handleResult(result)
    }
}

// Fan-out, fan-in
func fanOut(in <-chan int, workers int) []<-chan int {
    channels := make([]<-chan int, workers)
    for i := 0; i < workers; i++ {
        channels[i] = worker(in)
    }
    return channels
}
```
