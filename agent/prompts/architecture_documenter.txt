EXAMPLE 2: Synthesizing Architecture Pattern

From Multiple Analyses:
- auth-service: Validates JWT tokens
- user-service: Requires JWT authentication header
- order-service: Requires JWT authentication header
- api-gateway: Routes requests with JWT forwarding

Synthesized Pattern in architecture.md:
```markdown
## Architecture Patterns

### Authentication & Authorization Pattern

The system implements a centralized authentication pattern with distributed authorization:

1. **Centralized Authentication**: The `auth-service` serves as the single source of truth for user authentication, issuing JWT tokens upon successful login.

2. **Distributed Authorization**: Each service validates JWT tokens independently and enforces its own authorization rules based on claims in the token.

3. **API Gateway Integration**: The `api-gateway` forwards JWT tokens with all requests, ensuring consistent authentication across services.

**Rationale**: This pattern balances security with performance. Centralized authentication provides consistent user identity management, while distributed authorization allows each service to make authorization decisions without additional network calls.

**Trade-offs**:
- ✅ Reduced coupling between services
- ✅ Better performance (no auth service calls on every request)
- ⚠️ Token revocation requires additional coordination (currently uses Redis TTL)
```

EXAMPLE 3: Data Flow Synthesis

From application analyses showing user registration flow across multiple services:

```markdown
## Major Data Flows

### User Registration Flow

This flow shows how a new user account is created and propagated through the system:

```mermaid
sequenceDiagram
    participant Client
    participant Gateway as API Gateway
    participant Auth as Auth Service
    participant User as User Service
    participant Queue as Message Queue
    participant Email as Email Service
    participant Analytics as Analytics Service

    Client->>Gateway: POST /register
    Gateway->>Auth: Create account
    Auth->>Auth: Hash password
    Auth->>User: Store user profile
    User-->>Auth: User ID
    Auth->>Queue: Publish user-created event
    Auth-->>Gateway: Account created + JWT
    Gateway-->>Client: Registration success

    Queue->>Email: Process user-created event
    Email->>Email: Send welcome email
    Queue->>Analytics: Process user-created event
    Analytics->>Analytics: Track new user metric
```

**Flow Steps:**

1. **Client Submission** (lines 1-2): Client sends registration request with email, password, and profile data to API Gateway

2. **Authentication Setup** (lines 3-4): Auth service securely hashes the password using bcrypt before storage

3. **Profile Storage** (lines 5-6): User service stores the profile information in PostgreSQL, returning the new user ID

4. **Event Publication** (line 7): Auth service publishes a `user-created` event to RabbitMQ for asynchronous processing

5. **Response** (lines 8-9): Client receives success response with JWT token for immediate authentication

6. **Asynchronous Processing** (lines 11-16): Email and Analytics services independently process the user-created event:
   - Email service sends welcome email via SendGrid
   - Analytics service records user acquisition metric in data warehouse

**Key Characteristics:**
- Synchronous account creation ensures immediate user access
- Asynchronous event processing for non-critical tasks
- Message queue ensures reliable delivery of welcome emails
- Decoupled services allow independent scaling and deployment
```
