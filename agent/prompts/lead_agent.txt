**CRITICAL: Extracting External Dependencies**
When reading manifest files (Cargo.toml, package.json, pyproject.toml), extract ALL third-party
dependencies as structured objects — not bare strings. For each dependency:
1. **name**: The package name exactly as it appears in the manifest
2. **version**: The version constraint (e.g., "1.35", "^4.18.0", ">=3.0")
3. **category**: Classify by purpose using these categories:
   - `web-framework`: HTTP servers, routers (actix-web, axum, express, flask, fastapi)
   - `database`: Database drivers/ORMs (sqlx, diesel, prisma, sqlalchemy, mongoose)
   - `serialization`: Encoding/decoding (serde, prost, json, pydantic, protobuf)
   - `async-runtime`: Async execution (tokio, async-std, asyncio, gevent)
   - `crypto`: Cryptography, hashing (ring, openssl, bcrypt, jsonwebtoken)
   - `networking`: HTTP clients, gRPC, websockets (reqwest, hyper, grpc, axios)
   - `logging`: Logging/tracing (tracing, log4rs, winston, structlog)
   - `testing`: Test frameworks (criterion, proptest, jest, pytest)
   - `cli`: Command-line parsing (clap, structopt, commander, click)
   - `build-tool`: Build/codegen (tonic-build, prost-build, webpack)
   - `cloud-sdk`: Cloud provider SDKs (aws-sdk, google-cloud, azure)
   - `blockchain`: Chain interaction (ethers, web3, solana-sdk)
   - `monitoring`: Metrics/observability (prometheus, datadog, sentry)
   - `other`: Anything that doesn't fit the above
4. **purpose**: One sentence explaining what this package provides to the component

This structured data is essential for downstream architecture synthesis and technology inventory.

**PHASE 2: LIBRARY GRAPH & ANALYSIS**

Step 2.1: Build Library Dependency Graph
- Use Bash to run: python scripts/build_dependency_graph.py /tmp/{SERVICE_NAME}/service_discovery /tmp/{SERVICE_NAME}/dependency_graphs
- This script uses graph utilities (DependencyGraph, DependencyGraphBuilder) to:
  * Build library dependency graph with topological sort (Kahn's algorithm)
  * Compute analysis order by depth (depth 0 = no dependencies, depth 1+= increasing dependencies)
  * Generate JSON and markdown outputs for the library graph

The script outputs:
- /tmp/{SERVICE_NAME}/dependency_graphs/library_graph.json - Library dependency graph with depth-based analysis order
- /tmp/{SERVICE_NAME}/dependency_graphs/library_graph.md - Human-readable library graph visualization

Step 2.2: Verify Library Graph
- Use Read to load /tmp/{SERVICE_NAME}/dependency_graphs/library_graph.json
- Extract analysis_order with depth-based phases
- Verify all libraries are in the graph before proceeding

Library Graph Structure:
- Directed graph: library -> [library dependencies]
- Only LIBRARY-to-LIBRARY dependencies (internal libraries within same repo)
- Analysis order by depth:
  * depth 0 = libraries with no internal dependencies (analyze in parallel)
  * depth 1 = libraries depending only on depth 0 libraries (analyze in parallel after depth 0)
  * depth N = libraries depending on depth N-1 or lower libraries
- Max depth determines number of sequential analysis phases

Step 2.3: Analyze All Libraries by Depth
- For each depth level (0 to max_depth):
  * Get all libraries at that depth from analysis_order

  * Spawn code-library-analyzer subagent for each library at this depth (parallel within depth)

- Use Glob to find /tmp/{SERVICE_NAME}/service_analyses/*.md

<discovery_patterns>
**Rust Component Discovery with Glob and Read:**

Step 1: Find all Cargo.toml files
```
Use Glob with pattern: "Cargo.toml"
OR
Use Bash: find . -name "Cargo.toml" -not -path "*/target/*"
```

Step 2: For each Cargo.toml, extract info and classify:
```
Use Read to load file content
Parse for:
- Line starting with 'name = "' → extract package name
- Line starting with 'description = "' → extract description
- In [dependencies] section, lines with 'path = "' → internal dependencies
- Check for [[bin]] section or src/main.rs → APPLICATION
- Otherwise → LIBRARY
```

Example Cargo.toml (Application):
```toml
[package]
name = "auth-service"
description = "User authentication HTTP service"

[[bin]]
name = "auth-service"
path = "src/main.rs"

[dependencies]
common-utils = { path = "../common-utils" }  ← LIBRARY (internal)
database-models = { path = "../database-models" }  ← LIBRARY (internal)
serde = "1.0"  ← EXTERNAL
tokio = "1.35"  ← EXTERNAL
```

Classification:
- Type: APPLICATION (has [[bin]] or src/main.rs)
- Name: "auth-service"
- Libraries Used: ["common-utils", "database-models"]
- External Dependencies:
  [
    {"name": "serde", "version": "1.0", "category": "serialization", "purpose": "Serialize/deserialize data structures"},
    {"name": "tokio", "version": "1.35", "category": "async-runtime", "purpose": "Async runtime for concurrent I/O"}
  ]

Example Cargo.toml (Library):
```toml
[package]
name = "common-utils"
description = "Shared utility functions"

[dependencies]
serde = "1.0"  ← EXTERNAL
```

Classification:
- Type: LIBRARY (no [[bin]], no src/main.rs, only src/lib.rs)
- Name: "common-utils"
- Internal Dependencies: [] (no internal library dependencies)
- External Dependencies:
  [{"name": "serde", "version": "1.0", "category": "serialization", "purpose": "Serialize/deserialize data structures"}]

**Bash Commands for Discovery:**
```bash
# List all Cargo.toml locations
find . -name "Cargo.toml" -not -path "*/target/*"

# Quick package name check
grep -m 1 '^name = ' Cargo.toml | cut -d'"' -f2

# Check if it's an application (has main.rs)
test -f src/main.rs && echo "APPLICATION" || echo "LIBRARY"
```

<context_rules>
**CRITICAL: Direct Dependencies Only**

For Libraries:
If library C depends on [A, B], and B depends on [X]:
- When spawning analyzer for C, pass analyses of A and B ONLY
- Do NOT pass X's analysis, even though C transitively depends on X through B

For Applications:
If application MyApp uses libraries [Utils, Database], and Database depends on [Utils]:
- When spawning analyzer for MyApp, pass analyses of Utils and Database
- The analyzer can see that Database uses Utils by reading Database's analysis

Example:
```
common-utils (library, no deps)
    ↓
database-models (library) depends on [common-utils]
    ↓
auth-service (application) uses [common-utils, database-models]
    ↓ (runtime interaction)
user-service (application) - auth-service calls user-service APIs
```

When analyzing auth-service:
- Pass analyses of: common-utils, database-models (libraries it uses)
- Do NOT pass: user-service (that's a runtime interaction, discovered during analysis)

Why: This keeps context focused and manageable.
</context_rules>
