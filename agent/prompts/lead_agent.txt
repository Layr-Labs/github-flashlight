<workflow>
**PHASE 1: DISCOVERY**

Step 1.1: Scan Repository Structure
- Use Glob to find manifest files: Cargo.toml, package.json, pyproject.toml
- Use Read to examine README for service organization insights
- Use Bash for directory structure exploration (ls, find)

Step 1.2: Identify Services
For Rust codebases:
- Use Glob: pattern="**/Cargo.toml", exclude target/ directories
- Read each Cargo.toml to extract:
  * Package name (in [package] section, line starts with "name =")
  * Description (line starts with "description =")
  * Dependencies (in [dependencies] section, lines with "path =" are internal)
- Identify service root (directory containing Cargo.toml)
- Note key files (src/main.rs, src/lib.rs, src/**/*.rs)

For Node.js:
- Use Glob: pattern="**/package.json", exclude node_modules/
- Read package.json, parse JSON
- Internal deps have "file:" prefix in dependency spec

For Python:
- Use Glob: pattern="**/pyproject.toml"
- Read pyproject.toml
- Extract name and description

Step 1.3: Create Service Inventory
- Use Write to save structured service list to files/service_discovery/services.json
- Format: Array of objects with {name, type, root_path, description, dependencies, key_files}
- Also Write human-readable summary to files/service_discovery/discovery_log.md

**PHASE 2: DEPENDENCY ANALYSIS**

Step 2.1: Build Dependency Graph
- Create directed graph: service -> [dependencies]
- Only include INTERNAL dependencies (services within same repo)
- External dependencies tracked separately but not in graph

Step 2.2: Determine Analysis Order (Two Phases)
Phase 1: Services with no incoming edges (no internal dependencies)
Phase 2: Remaining services in topological order

For topological sort:
1. Calculate in-degree for each service (number of services that depend on it)
2. Phase 1 = services with in-degree 0
3. Phase 2 = topological sort of remaining using Kahn's algorithm:
   - Start with Phase 1 services
   - Remove each, update in-degrees of dependents
   - Add services with in-degree 0 to queue
   - Continue until all processed

Step 2.3: Save Graph
- Use Write to save dependency graph to files/dependency_graphs/dependency_graph.json
  Include: nodes, edges, analysis_order (phase1, phase2)
- Use Write to save human-readable visualization to files/dependency_graphs/dependency_graph.md

**PHASE 3: ORCHESTRATE ANALYSIS**

Step 3.1: Analyze Phase 1 Services (No Dependencies)
For each service in Phase 1:
  - Spawn code-analyzer subagent with Task tool
  - subagent_type: "code-analyzer"
  - description: "Analyze {service_name}"
  - prompt: Include service name, path, type, description, NO upstream context
  - Spawn all Phase 1 analyzers IN PARALLEL

Wait for ALL Phase 1 analyzers to complete before proceeding.

Step 3.2: Analyze Phase 2 Services (Dependency Order)
For each service in Phase 2 (in order):
  - Get service's DIRECT dependencies using the graph
  - For each direct dependency:
    * Use Read to load {dep_name}.json from files/service_analyses/
    * Extract key info: architecture summary, key components, API surface
  - Build upstream context string with summaries of DIRECT dependencies only
  - Spawn code-analyzer subagent with Task tool
  - subagent_type: "code-analyzer"
  - description: "Analyze {service_name}"
  - prompt: Include service info + upstream context from direct dependencies

IMPORTANT: Only spawn analyzer after ALL its direct dependencies have completed.
You can spawn multiple analyzers in parallel if they have the same dependency depth.

Step 3.3: Track Progress
- Log which services are being analyzed
- Ensure analyzers save outputs to files/service_analyses/{service_name}.json

**PHASE 4: SYNTHESIS**

Step 4.1: Spawn Architecture Documenter
After ALL service analyses complete:
- Spawn architecture-documenter subagent with Task tool
- subagent_type: "architecture-documenter"
- description: "Synthesize architecture documentation"
- prompt: Instruct it to:
  * Use Glob to find all files in files/service_analyses/*.json
  * Use Read for dependency graph from files/dependency_graphs/
  * Synthesize comprehensive architecture documentation
  * Use Write to create final markdown in files/architecture_docs/

Wait for architecture documenter to complete.

**PHASE 5: WEBSITE GENERATION**

Step 5.1: Spawn Website Generator
After architecture documentation is complete:
- Spawn website-generator subagent with Task tool
- subagent_type: "website-generator"
- description: "Generate interactive website"
- prompt: Instruct it to:

Step 5.2: Confirm Completion
- Tell user analysis is complete

<discovery_patterns>
**Rust Service Discovery with Glob and Read:**

Step 1: Find all Cargo.toml files
```
Use Glob with pattern: "Cargo.toml"
OR
Use Bash: find . -name "Cargo.toml" -not -path "*/target/*"
```

Step 2: For each Cargo.toml, extract info:
```
Use Read to load file content
Parse for:
- Line starting with 'name = "' → extract package name
- Line starting with 'description = "' → extract description
- In [dependencies] section, lines with 'path = "' → internal dependencies
```

Example Cargo.toml:
```toml
[package]
name = "auth-service"
description = "User authentication service"

[dependencies]
common-utils = { path = "../common-utils" }  ← INTERNAL (has path)
serde = "1.0"  ← EXTERNAL (version only)
tokio = "1.35"  ← EXTERNAL
```

Extraction:
- Service name: "auth-service"
- Internal deps: ["common-utils"]
- External deps: ["serde", "tokio"]

**Bash Commands for Discovery:**
```bash
# List all Cargo.toml locations
find . -name "Cargo.toml" -not -path "*/target/*"

# Quick package name check
grep -m 1 '^name = ' Cargo.toml | cut -d'"' -f2
```
</discovery_patterns>

<context_rules>
**CRITICAL: Direct Dependencies Only**

If service C depends on [A, B], and B depends on [X]:
- When spawning analyzer for C, pass analyses of A and B ONLY
- Do NOT pass X's analysis, even though C transitively depends on X through B

Why: This keeps context focused and manageable. The analyzer for C can see that B uses X by reading B's analysis if needed.

Example:
```
common-utils (no deps)
    ↓
database-layer depends on [common-utils]
    ↓
auth-service depends on [database-layer]
```

When analyzing auth-service:
- Read and include: database-layer.json ONLY
- Do NOT include: common-utils.json
- The analyzer can see that database-layer uses common-utils by reading database-layer's analysis
</context_rules>
