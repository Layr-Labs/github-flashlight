**KIND-SPECIFIC ANALYSIS**

Based on the component kind ({component_kind}), focus your analysis appropriately:

- **library**: Focus on public API surface, reusable patterns, and how other components would consume this.
- **service**: Focus on endpoints, request handling, runtime behavior, external system integrations.
- **cli**: Focus on command structure, argument parsing, user-facing behavior, exit codes.
- **contract**: Focus on interface definitions, state management, security constraints, gas/cost considerations.
- **infra**: Focus on resource definitions, dependencies between resources, deployment topology.
- **pipeline**: Focus on data flow stages, scheduling, failure handling, data transformations.
- **frontend**: Focus on component hierarchy, state management, API consumption, user interactions.

**ANALYSIS STRUCTURE**

Perform deep analysis covering:

1. **Architecture**: Overall design patterns and structure
2. **Key Components**: Major modules, structs, classes, functions (5-12 items)
3. **Data Flows**: Important data paths and control flows (with Mermaid diagrams)
4. **External Dependencies** (CRITICAL — see detailed instructions below)
5. **Internal Dependencies**: HOW this component uses its direct dependencies (from upstream context)
6. **API Surface**: Public interfaces exposed to other components or users

For services/applications, also include:
7. **External Systems**: Infrastructure and cloud services this component connects to at runtime
8. **Component Interactions**: Calls to other components in this codebase (HTTP, gRPC, shared DB, etc.)

Steps:
1. Read the manifest file (Cargo.toml, package.json, pyproject.toml, go.mod) to get the

2. For EACH external dependency, document:
   - **Name**: Exact package name
